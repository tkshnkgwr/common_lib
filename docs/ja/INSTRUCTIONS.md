# AI開発・コーディング指示書 (INSTRUCTIONS.md) - common_lib

[English](../en/INSTRUCTIONS.md) | **日本語版**

本ドキュメントは、AIエージェントおよび開発者が `common_lib` プロジェクトに対してコードの追加・修正を行う際に遵守すべきコーディング規約、実装方針、および作業時のルールについて定めたものです。

---

## 1. 命名規則

Rustの標準的な命名規則（Idiomatic Rust Style）に従います。

- **変数・関数・マクロ**: スネークケース (`snake_case`)
  - 例: `check_single_instance`, `suggest_tags`, `raw_bytes`
- **型・構造体・列挙型・トレイト**: パスカルケース (`PascalCase`)
  - 例: `SingleInstanceGuard`, `Error`, `DiffType`, `DiffPart`
- **定数・スタティック変数**: スクリーミングスネークケース (`SCREAMING_SNAKE_CASE`)
  - 例: `MAX_BUFFER_SIZE`
- **ファイル名・ディレクトリ名**:
  - ソースコードはスネークケース (`snake_case.rs`)
  - ドキュメントファイルは大文字のスネークケース (`DOC_NAME.md`)
    - 例: `ARCHITECTURE.md`, `SPEC.md`
  - ドキュメントの多言語ディレクトリ構成: `docs/en/` および `docs/ja/` 配下にサフィックス（`_JA` 等）を付けずに配置
- **テスト関数名**:
  - `test_` で開始し、テスト対象の関数名と検査対象を明示するスネークケース。
  - 例: `test_acquire_single_instance`, `test_count_occurrences`

## 2. エラーハンドリング方針

パニック（`panic!`, `unwrap`）を避けることを最優先とし、エラーは必ず戻り値（`Result`）として返却して呼び出し側で処理させます。

- **共通エラー型の使用**:
  - クレート共通のエラー型 `crate::Error` (列挙型) および結果型の別名 `crate::Result<T>` を使用します。
  - 新たな種類のエラーが発生しうる機能を実装する場合は、`src/error.rs` の `Error` 列挙型に新規バリアントを追加し、`Display` トレイトの実装で分かりやすいエラーメッセージを定義してください。
- **プラットフォーム非依存の実装**:
  - Windows API 等に起因するエラーは、プラットフォーム固有のブロック内で適切に補足し、共通の `Error` 型（例: `Error::MutexCreation`）に変換して返します。

## 3. コンポーネントやモジュールの分割基準

コードは機能面での結合度およびプラットフォーム依存性に基づき、以下のように配置および分割します。

- **OS / プラットフォーム依存の機能**:
  - デスクトップアプリ固有、またはOSのAPIに密接に依存する機能は `src/desktop.rs` に配置します。
  - 外部APIへの依存が発生する場合は、`#[cfg(target_os = "windows")]` やフィーチャーフラグ `#[cfg(feature = "windows_desktop")]` などを用いて対象を分離します。
  - Windows 以外のOSや、フィーチャーフラグがオフのビルド環境でもコンパイルエラーを起こさないよう、同名のダミー構造体およびダミー関数（スタブ）を常に提供してください。
- **プラットフォーム非依存の純粋ロジック**:
  - 文字列検索や差分計算など、プラットフォームに依存しない機能は `src/text.rs` に配置します。
  - データのシリアライズなどが必要な場合は、標準的な外部クレート（`serde` など）に依存を留めます。
- **新しいモジュールの追加**:
  - 機能規模が拡大し、新たなドメイン（例: ネットワーク、ファイルI/O）が発生した場合は、`src/` 配下に新規モジュールファイルを作成し、`src/lib.rs` で再エクスポートします。
- **コードの規模とリファクタリング基準**:
  - 1つのソースコードファイル（`.rs`）が 1,000 行を超えた場合は、保守性や可読性の低下を防ぐため、機能やサブモジュールごとにファイルを分割・リファクタリングすることを強く推奨します。Gemは 1,000 行を超えたファイルを検知した場合、リファクタリングを提案しなければなりません。

## 4. AIが出力する際のフォーマット指定

AIエージェント（Gem）がユーザーに対して提案や回答を行う際は、以下のフォーマットとルールを遵守してください。

- **簡潔な解説**:
  - コード変更の背景や意図は最小限の記述に留め、変更内容そのもの（コードブロックや diff）を明確に提示してください。
- **ファイルパスの表記**:
  - ファイルやディレクトリへの言及は、必ず `file://` スキームを用いた絶対パスかつリンクテキストにバックティックスを付けないクリック可能なMarkdownリンク形式としてください。
  - 良い例: `[lib.rs](file:///c:/Users/632792/Documents/自作/common_lib/src/lib.rs)`
  - 悪い例: ``[`lib.rs`](file://...)`` (バックティックスを付けるとリンクが無効になります)
- **変更手順の明確化**:
  - 計画やタスクを提案する際は、実施対象ファイルをコンポーネント（依存性の低い順）ごとに構造化して記述してください。
