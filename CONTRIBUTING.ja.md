# common_lib への貢献ガイドライン

`common_lib` に関心を持っていただき、ありがとうございます！ バグ報告、機能の提案、およびコードの提供を歓迎します。

円滑な共同開発を進めるために、以下のガイドラインに従ってください。

## 1. ローカル開発フロー

### 前提条件
Rust (stable, 2024 エディション以降) がインストールされていることを確認してください。

### コミット前・プッシュ前の検証プロセス
変更をコミットまたはプルリクエストとして送信する前に、必ずローカル環境で以下の検証パイプラインを実行し、エラーや警告がゼロであることを確認してください。

1. **ユニットテストの実行**:
   ```bash
   cargo test
   ```
2. **Clippy (静的解析ツール) の実行**:
   コードが Rust の推奨されるスタイルに準拠しており、警告がないことを確認します。
   ```bash
   cargo clippy --all-targets -- -D warnings
   ```
3. **Rustfmt (フォーマッタ) の確認**:
   コードのフォーマットが Rust 標準のガイドラインに厳密に準拠しているか確認します。
   ```bash
   cargo fmt --check
   ```

---

## 2. コーディングルール
- **ユニットテストの追加**: 新機能の追加やコアロジックの修正を行う場合は、必ず関連するユニットテスト（通常はファイル下部の `tests` モジュール）を追加・拡張してください。
- **ドキュメンテーション**: すべてのパブリック API (`pub fn`, `pub struct`, `pub enum` 等) には、ドキュメンテーションコメント (`///`) を記述してください。また、APIの仕様に変更があった場合は `docs/SPEC.md` や `docs/DIAGRAM.md` を忘れずに更新してください。
- **設定ファイル・CI定義の保護**: `.github/workflows/` 下のワークフロー定義、`.github/dependabot.yml`、`.editorconfig` などを変更する場合は、必ず `CHANGELOG.md` にその変更内容と理由を記録してください。

---

## 3. 貢献手順

### ステップ 1: Issue の作成 / 提案
バグを発見した場合や新機能を提案したい場合は、まず Issue を作成して議論を行ってください。

### ステップ 2: ブランチの作成
変更作業用のブランチを作成します。
```bash
git checkout -b feature/your-feature-name
```

### ステップ 3: 実装とドキュメントの作成
コードの実装、テストコードの追加、および必要に応じたドキュメントの更新を行います。

### ステップ 4: CHANGELOG.md への記録
すべての貢献において、`CHANGELOG.md` への変更内容の記録が必要です。[Keep a Changelog](https://keepachangelog.com/en/1.0.0/) 形式に従い、以下のカテゴリに分類して記述してください：
- `Added`: 新機能の追加
- `Fixed`: バグの修正
- `Optimized`: パフォーマンス改善やリファクタリング
- `Removed`: 非推奨機能の削除

### ステップ 5: プルリクエストの作成
リモートリポジトリに変更をプッシュし、プルリクエストを作成します。CI が通り、コードレビューが完了した後にマージされます。
