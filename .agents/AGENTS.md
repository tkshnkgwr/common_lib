# AI Agent Instructions for Common Utility Library (common_lib)

本プロジェクトにおいて、AIエージェント（以降「Gem」と呼ぶ）が従うべき自動ドキュメント生成、品質管理、および対話・開発プロセスに関する指示書です。

## 1. 基本設定と対話ガイドライン
- **呼称**: ユーザーはAIを「Gem」と呼びます。Gem自身もユーザーに対してGemとして振る舞い、簡潔・丁寧かつ分かりやすいサポートを行います。
- **回答フォーマット方針**:
  - **結論ファースト**: 冒頭で結論や成果の概要を簡潔に示します。
  - **背景・理由の明記**: コード変更や技術的判断を行った場合は、その根拠と影響範囲を明確に記述します。
  - **視覚的整理**: GitHubスタイルの Markdown（リスト、表、太字、ファイルリンク `file:///...`）を積極的に活用します。
- **技術解説・対話方針（他言語経験に配慮した解説）**:
  - ユーザーは他言語での開発経験をお持ちですが、**Rust固有の言語仕様・専門用語（所有権、ライフタイム、トレイト、マクロ等）には不慣れ**です。
  - そのため、修正内容や設計の説明を行う際は、初心者向けにかみ砕きすぎる必要はありませんが、これら特定の言語固有の難解な専門用語をそのまま多用せず、一般的・共通的なプログラミング概念（参照、インターフェース、モジュール、関数など）に置き換えるか、直感的に理解できるよう分かりやすく補足・解説を添えること。
- **Markdown 表（テーブル）のフォーマット＆コンパクト化ルール**:
  - Markdown内の表（テーブル）を作成・更新する際は、パイプ位置を揃え、セルの余白や表示幅を考慮して視認性を高めるとともに、不要な改行を抑えてコンパクトで読みやすい構造を維持すること。
- **Git コミットメッセージ規約**:
  - コミットメッセージは以下の Conventional Commits 形式に従うこと：
    - `feat:` 新機能追加
    - `fix:` バグ修正
    - `docs:` ドキュメントの変更
    - `refactor:` リファクタリング
    - `perf:` パフォーマンス最適化
    - `test:` テストの追加・修正
    - `chore:` ビルド・維持管理タスク
- **大規模変更・破壊的変更前の事前承認ルール**:
  - モジュール構成の刷新、APIの互換性破壊、アルゴリズムの大きな変更を行う場合は、直接実装に入る前に**実装計画（プラン）**を作成・提示し、ユーザーの事前承認を得てから作業を開始します。
- **目的**: プロジェクトのソースコードや設計の変更と、ドキュメント（Markdownファイル）の内容が常に一致し、乖離しないようにします。

## 2. 自動ドキュメンテーションと更新ルール
コードの変更、リファクタリング、機能の追加、バグの修正、または依存ライブラリの更新などを行った場合、Gemは必ず以下のドキュメントファイル群を検査し、必要に応じて自動的に作成または更新しなければなりません。
ドキュメントを更新する際は、必ず **英語版と日本語版の両方のファイル** を更新し、内容が完全同期されていることを保証します。
**※ただし、Markdownファイル（`.md`）のみの変更・修正の場合は、自動ドキュメンテーションおよび更新ルールは適用外とし、CHANGELOGやその他のドキュメントの自動更新は行いません。**

### ドキュメントの命名規則と多言語化
- **構造**: `docs/en/` (英語版) および `docs/ja/` (日本語版) の2つのフォルダを作成し、同一ファイル名で管理します（`_JA` 等のサフィックスは付加しません）。
  - *例*: `docs/en/SPEC.md` と `docs/ja/SPEC.md`
- **命名規則**: すべてのドキュメントファイル名（Markdownファイル等）は、**大文字のスネークケース（拡張子は `.md`）**に統一します。
  - *例*: `SPEC.md`, `ARCHITECTURE.md`, `INSTRUCTIONS.md`, `TODO.md`
- **言語切替リンク**: 各ドキュメント（`docs/en/` および `docs/ja/` 内）の最上部には、もう一方の言語版への相互切り替えリンクを必ず配置します。
  - *例 (英語版の先頭)*: `**English** | [日本語版](../ja/SPEC.md)`
  - *例 (日本語版の先頭)*: `[English](../en/SPEC.md) | **日本語版**`

### 対象ファイルと更新ポリシー

#### 1. [CHANGELOG.md (EN)](../docs/en/CHANGELOG.md) / [CHANGELOG.md (JA)](../docs/ja/CHANGELOG.md)
- **役割**: プロジェクトの変更履歴の記録。ルートの `CHANGELOG.md` には案内リンクのみを配置し、履歴の実体は `docs/en/` および `docs/ja/` に記録します。
- **更新ルール**:
  - 実装が完了するたびに、実施した変更内容を追記します。
  - セクションは日付単位 `## [YYYY-MM-DD]` で区切り、最新のものが一番上に来るようにします。
  - 変更内容を以下のカテゴリに分類して記述します：
    - `### Added`: 新機能の追加
    - `### Fixed`: バグの修正
    - `### Optimized`: パフォーマンス改善、コード最適化、リソース使用量の削減など
    - `### Removed`: 機能の削除

#### 2. [SPEC.md (EN)](../docs/en/SPEC.md) / [SPEC.md (JA)](../docs/ja/SPEC.md)
- **役割**: ライブラリの機能仕様および技術スタックの定義。
- **更新ルール**:
  - 提供するAPI、モジュール構成、対応OS（Windows/クロスプラットフォーム）、依存関係などの仕様に変更が生じた場合、常に最新の仕様に書き換えます。

#### 3. [DIAGRAM.md (EN)](../docs/en/DIAGRAM.md) / [DIAGRAM.md (JA)](../docs/ja/DIAGRAM.md)
- **役割**: システム構成、データフロー、コンポーネント構造の可視化。
- **更新ルール**:
  - モジュール構成、二重起動防止などのシーケンス、差分計算などのデータフロー、外部依存APIとの関係などに変更が生じた場合、Mermaidダイアグラムを更新して最新の構成・フローを反映させます。

#### 4. [README.md](../README.md) / [README_JA.md](../README_JA.md)
- **役割**: プロジェクトの概要、主な機能、ビルド・使用手順、設定の解説。
- **更新ルール**:
  - APIの利用方法、動作要件（RustのバージョンやOS等）に変更があった場合、英語（`README.md`）および日本語（`README_JA.md`）の両ファイルを更新します。
  - 両ファイルの上部には多言語版（日本語版/英語版）への相互リンク、および各種ステータスバッジを適切に維持すること。また、`docs/en/` または `docs/ja/` 配下の各ドキュメントへのリンク一覧を維持すること。

#### 5. [FOOTPRINTS.md (EN)](../docs/en/FOOTPRINTS.md) / [FOOTPRINTS.md (JA)](../docs/ja/FOOTPRINTS.md)
- **役割**: パフォーマンスやバイナリサイズ、リソース使用量（CPU、メモリ等）の計測記録。
- **更新ルール**:
  - リリリースビルドのバイナリサイズ変更、メモリフットプリントの大幅な増減、ビルド時の最適化設定（strip設定など）を変更した場合、または新規のフットプリント計測結果が得られた場合に、本ドキュメントを更新または追記します。

#### 6. [ARCHITECTURE.md (EN)](../docs/en/ARCHITECTURE.md) / [ARCHITECTURE.md (JA)](../docs/ja/ARCHITECTURE.md)
- **役割**: システムの構成、アーキテクチャ設計、採用技術スタック、およびモジュール間連携の解説。
- **更新ルール**:
  - 主要なモジュールの追加、依存パッケージの変更、またはアーキテクチャレベルでの変更が生じた場合、常に最新のシステム構造に適合するように設計内容を更新します。

#### 7. [INSTRUCTIONS.md (EN)](../docs/en/INSTRUCTIONS.md) / [INSTRUCTIONS.md (JA)](../docs/ja/INSTRUCTIONS.md)
- **役割**: AIエージェントおよび開発者がコードを修正・追加する際に遵守すべき開発・コーディング指示書。
- **更新ルール**:
  - コーディングスタイルやフォーマットの指示、命名規則に変更が生じた場合、または新たな開発上の規約が追加された場合に内容を更新します。

#### 8. [TODO.md (EN)](../docs/en/TODO.md) / [TODO.md (JA)](../docs/ja/TODO.md)
- **役割**: プロジェクトの実装状況（Done）と、直近のタスク（Todo）、および将来の拡張機能（Backlog）の管理。
- **更新ルール**:
  - 新機能の実装完了、新規タスクの追加、またはロードマップの変更が生じた場合、開発状況の「現在地」を正しく表現するように更新します。

#### 9. [TESTING.md (EN)](../docs/en/TESTING.md) / [TESTING.md (JA)](../docs/ja/TESTING.md)
- **役割**: テスト方針、検証項目、テスト実行手順、事前検証プロセスのガイド。
- **更新ルール**:
  - テスト観点の変更や新たな検証コマンドの導入があった場合に最新化します。

#### 10. [RELEASE.md (EN)](../docs/en/RELEASE.md) / [RELEASE.md (JA)](../docs/ja/RELEASE.md)
- **役割**: バージョン更新およびリリース作業の手順書。
- **更新ルール**:
  - リリリースプロセスやビルド手順、タグ打ち手順に変更があった場合に更新します。

#### 11. [CONTRIBUTING.md (EN)](../docs/en/CONTRIBUTING.md) / [CONTRIBUTING.md (JA)](../docs/ja/CONTRIBUTING.md)
- **役割**: 開発への参加方法およびローカル検証プロセスの提示。
- **更新ルール**:
  - テスト・静的解析の実行方法や、開発フローに変更が生じた場合に更新します。

#### 12. [SECURITY.md (EN)](../docs/en/SECURITY.md) / [SECURITY.md (JA)](../docs/ja/SECURITY.md)
- **役割**: セキュリティポリシーおよび安全な実装のガイドライン。
- **更新ルール**:
  - 二重起動防止に関するセキュリティ考慮事項に変更があった場合、または脆弱性報告先等に更新があった場合に最新の状態に保ちます。

#### 13. [EXAMPLES.md (EN)](../docs/en/EXAMPLES.md) / [EXAMPLES.md (JA)](../docs/ja/EXAMPLES.md)
- **役割**: 各APIの実用例を記述したクックブック。
- **更新ルール**:
  - 新機能の追加や既存APIの仕様変更時に、ユーザーが使い始められるようにコード例を追記・更新します。

#### 14. Rustdoc (ソースコード内のドキュメントコメント `///` / `//!`)
- **役割**: APIの利用方法および技術的詳細のインライン解説。
- **更新ルール**:
  - APIのシグネチャ変更、新しい構造体・関数・モジュールの追加、または既存ロジックの動作仕様が変更された場合、必ずソースコード内の対応するドキュメントコメント（`///` や `//!`）を最新の仕様に書き換えます。

## 3. 品質管理・自動テストと事前検証ルール
AIがソースコードを変更した際、あるいはタスクを完了する前には、必ず以下のルールに従って品質管理と動作検証を行わなければなりません。

- **ユニットテストの追加・拡充**:
  - 新機能の追加やコアロジックの変更を行う場合、可能な限り関連するユニットテストコード（`src/` 内のモジュールテスト等）を追加・拡張すること。
- **コード規模とリファクタリング（モジュール分割）**:
  - 単一のソースファイル（例: `src/lib.rs` 等）が **1,000 行** を超えたあたりで、Gemは関心事・役割に応じたモジュール分割リファクタリングを検討・提案し、適切なディレクトリ・モジュール構造へ移行させること。
- **ローカルでの事前検証プロセス**:
  - コミットまたはタスクの完了報告を行う前に、ローカル環境で以下のコマンドを順次実行し、すべて合格（エラー・警告ゼロ）することを確認すること：
    1. `cargo test` （テストの合格）
    2. `cargo clippy --all-targets -- -D warnings` （Clippyの警告・エラーゼロ）
    3. `cargo fmt --check` （コードフォーマットの完全準拠）
    4. `cargo doc --no-deps --document-private-items` （Rustdocビルドエラー・警告ゼロ）
  - **※ただし、Markdownファイル（`.md`）のみの変更・修正の場合は、これらの事前検証プロセス（テスト、Clippy、フォーマット、ドキュメントのビルド等）はすべてスキップして構いません。**

## 4. 開発および実行手順
1. **コード変更の検知**: コードの追加・変更・削除を伴うタスクの完了時に、影響を受ける範囲を特定する。
2. **整合性チェック**: 変更点が各ドキュメント（SPEC.md, README.md 等）およびCI/CD、テストコードに影響を与えるか評価する。
3. **ローカル検証の実行**: 「3. 品質管理・自動テストと事前検証ルール」に従い、ローカルでの検証コマンドがすべて合格することを確認する。
4. **自動更新**: 影響を受けるドキュメントに対して変更内容を反映する（特に `CHANGELOG.md` への追記は必須）。
5. **確認と報告**: 更新したファイルとローカルテストの実行結果を、ユーザーへの作業完了報告（walkthrough等）で明示する。
